# Clax Language Specification

**Version 2.0 (draft)**

Clax is a minimal extension to the C programming language that adds class
functionality. Clax source is converted to plain C by a transpiler; the
generated C is then compiled with a normal C toolchain. Clax adds classes and a
small object model on top of C — everything else is ordinary C and passes
through unchanged.

This document supersedes V1. Changes from V1 are summarized in Section 13.

---

## 1. Design Principles

1. **Stay C-like.** Clax is C plus classes. Anything that is not class syntax is
   plain C and is left unchanged by the transpiler.
2. **One class per file.** A class is defined in its own file; the file name is
   the class name (`MyClass.cx` defines `MyClass`).
3. **No class headers.** The programmer never writes a C header for a class. The
   transpiler generates all C headers.
4. **Objects are heap-allocated references.** Every object is a pointer to a
   heap allocation. There is no stack/value object form. An object variable,
   parameter, member, or return value is always a reference.
5. **One creation path, one destruction path.** Every object is born from `new`
   and dies on `delete`.
6. **Caller owns returned and cloned objects.** Any object handed back from a
   method (including `clone`) is owned by the receiver and must be `delete`d by
   it.
7. **The transpiler is not a compiler.** Its only job is Clax to C. The C
   toolchain compiles the generated C.
8. **Single command, single project directory.** The transpiler is one
   C-language command. All project-owned `.c` and `.cx` files live in one
   directory. Reusable classes may live outside the project directory and are
   pulled in via `--include` search paths (see Section 14).

## 2. Files and Project Layout

| Extension | Contents |
|-----------|----------|
| `.cx`     | A class file (one `class` block) **or** a program file (ordinary C plus class includes, including `int main()`). |
| `.c`      | Ordinary C source. Passes through to the C compiler unchanged. |
| `.hx`     | **Only one such file is permitted per project, named `project.hx`** (Section 2.2). Plain C used for project-wide constants, macros, and typedefs. |

A Clax project may not contain hand-written `.h` files. Class headers are
generated by the transpiler, and project-wide declarations belong in
`project.hx`.

A project's own `.c` and `.cx` files reside in a **single directory** — the
*project directory*. Reusable classes (typically the standard library and
shared utilities) may live in separate directories registered with the
transpiler via `--include` (Section 14). All transpiler output for a project
is written into one `generated_c/` subdirectory of the project directory.

The programmer writes `int main()` themselves in whichever `.cx` (or `.c`)
file they choose; nothing is auto-generated as an entry point.

### 2.1 `#include` and include search

Class files reference each other through `#include`. Two forms are supported:

| Form | Resolution |
|------|------------|
| `#include "X.cx"` or `#include "./X.cx"` | Relative to the directory of the file doing the including. Used for project-local classes. |
| `#include <X.cx>` | Searched in the order of `--include` directories given on the command line. Used for reusable classes. |

Both forms are rewritten to `#include "X.h"` in the generated C; the path is
flattened so every generated header sits beside its peers in `generated_c/`.
A `.cx` include that cannot be resolved is a transpile-time error.

Plain C includes (`<stdio.h>`, `"my_helper.h"`, etc.) pass through unchanged.

### 2.2 `project.hx` — the one allowed project header

Each project has exactly one hand-written header file, named `project.hx`, in
the project directory. It holds project-wide constants, macros, and typedefs
that every translation unit in the project should see. Its contents are
ordinary C; the `.hx` extension distinguishes it from generated headers and
makes it clear at a glance that it is not a `.cx` class file.

The transpiler manages it as follows:

- If `<project-dir>/project.hx` does not exist, an empty stub is created on
  first run.
- Its contents are copied verbatim into `<project-dir>/generated_c/project.h`
  on every transpile.
- The generated `clax_runtime.h` `#include`s `"project.h"`, and every
  generated class header in turn includes `clax_runtime.h`, so the contents
  of `project.hx` are visible in every generated `.c` file.

A typical `project.hx` looks like:

```c
#ifndef PROJECT_HX
#define PROJECT_HX

#define MAX_PLAYERS    8
#define DEFAULT_PORT   7777

typedef unsigned long long Tick;

#endif
```

No other hand-written `.h` files are permitted; multiple project headers
defeat the goal of a flat, transpiler-managed project layout. If you find
yourself wanting another header, the contents belong either in `project.hx`
or in a `.cx` class.

## 3. Class Definition

A class is declared with `class`, the class name (matching the file name), a
body, and a terminating `};`.

```c
// MyClass.cx
class MyClass {
    int x, y;                          // data members

    MyClass(int x, int y) {            // constructor
        this::x = x;
        this::y = y;
    }

    ~MyClass() {                       // destructor
        // cleanup
    }

    void add(MyClass other) {          // method (objects are references)
        this::x += other::x;
        this::y += other::y;
    }

    int getX() {
        return this::x;
    }
};
```

### 3.1 Data members

Data members may be any C type, **or a class type** (new in V2 — see Section 6).
A class-typed member is a reference (pointer); see Section 6 for its lifecycle.

### 3.2 Constructor

- Named exactly like the class; no return type.
- Runs when an object is created with `new`.
- Exactly one constructor per class (no overloading).
- The transpiler injects, as the constructor's first actions, code that writes
  the object's hidden header (Section 7).

### 3.3 Destructor

- Named like the class, prefixed with `~`; no return type, no parameters.
- Runs **only** on explicit `delete`; never automatically.
- Freeing any C memory or class-typed members the object owns is the
  destructor's responsibility, written as ordinary Clax/C code.

### 3.4 Methods

- Declared like C functions inside the class body.
- May take C types, class types, and `Object` (Section 8) as parameters.
- May return C types **or** class types or `Object` (new in V2). A returned
  object is owned by the caller (principle 6).

### 3.5 Methods as callbacks — `callable()`

A method cannot be passed directly as a C function pointer: the transpiler
injects `ClassName* this` as its first parameter (Section 13.4), so its signature
never matches a plain callback type such as `void (*)(App*, void*, void*)`.

`callable(recv::method)` bridges the gap. It evaluates to the **address of a
trampoline** the transpiler generates for `method` — an ordinary C function, with
no `this`, that can be stored and invoked as a plain function pointer:

```
button_cb::set_callback(SDL_MOUSEBUTTONDOWN, callable(this::on_click));
```

`recv` may be `this`, a class-typed variable, or a class name
(`callable(Button::on_click)`).

**The receiver travels through an `object` parameter.** A trampoline is emitted
only for a method that declares a parameter named `object`. The trampoline has
that method's parameter list (without the injected `this`); it casts `object` to
the class and calls the method with it as `this`, so the body uses `this::`
normally:

```
// Clax — a genuine method; this::rect needs no cast
void on_click(App app, void *object, void *data) {
    SDL_Rect r = this::rect;
    ...
}

// Generated: the method, plus its trampoline (Section 13.4)
void Button__on_click(Button* this, App* app, void* object, void* data) { ... }
void Button__on_click__cb(App* app, void* object, void* data) {
    Button__on_click((Button*)object, app, object, data);
}
```

The caller must arrange for the bound instance to arrive in the `object` slot;
an event system typically stores the object when the handler is registered and
passes it back on dispatch. A method with no `object` parameter gets no
trampoline, and `callable()` on it fails to compile.

## 4. The `::` Operator

`::` is the **only** operator for accessing class members and methods, and is
the marker the transpiler uses to recognize class operations.

- `this::member` — a member of the current object, inside a method.
- `obj::member` — a member of an object.
- `obj::method(args)` — a method call on an object.
- `callable(obj::method)` — the *address* of a method as a plain C function
  pointer, for use as a callback (Section 3.5).

`.` and `->` are never used on Clax objects; they remain available for ordinary
C use (C structs, pointers). `::` accesses exactly one member; anything after it
is plain C (`this::buf[i]` lowers to `this->buf[i]`).

Three method-like operations are built into **every** object and are also
invoked with `::` — `typeof`, `clone`, and the implicit lifecycle. These are
described in Sections 9 and 10.

## 5. Object Lifecycle

### 5.1 Creation

An object is created with `new`, only as the initializer of a standalone,
single-variable declaration:

```c
MyClass obj = new MyClass(1, 2);
```

`new` may not appear inside an expression and may not be combined with
multi-variable declarations.

### 5.2 Use

```c
obj::add(other);
int v = obj::getX();
```

### 5.3 Destruction

```c
delete obj;
```

`delete` runs the destructor, then frees the object. `delete` on a `NULL`
object is a safe no-op (Section 11). Forgetting `delete` leaks the object.

### 5.4 Object parameters and assignment

Objects are **always references**. Passing an object to a method passes the
reference — the callee sees the same object, not a copy. Assigning one object
variable to another copies the reference, not the object; both then refer to
the same heap object. To obtain an independent copy, use `clone` (Section 10).

---

## 6. Classes Containing Classes

A class member may be of a class type (new in V2):

```c
// Container.cx
#include "./MyClass.cx"

class Container {
    MyClass item;          // a class-typed member: a reference (MyClass*)
    Container() {
        this::item = NULL; // class-typed members start as NULL by default
    }
    ~Container() {
        delete this::item; // owner is responsible; delete NULL is safe
    }
    void setItem(MyClass m) {
        this::item = m;
    }
};
```

Rules:

- A class-typed member is a **reference** (`MyClass*` in generated C).
- A class-typed member is **`NULL`** until explicitly assigned. The transpiler
  zero-initializes it; the constructor may set it.
- Clax does **not** automatically construct or destroy class-typed members. The
  containing class's constructor/destructor must `new`/assign and `delete` them
  as the programmer sees fit. (`delete NULL` is safe, so an unset member is
  harmless to `delete`.)
- A class file that uses another class type must `#include` that class's `.cx`
  file. The transpiler resolves includes transitively and emits struct
  definitions in dependency order.
- Recursive containment via references is permitted (a class may contain a
  member of its own type), because members are pointers. Direct value
  containment does not exist, so there is no infinite-size problem.

## 7. Object Header and Memory Layout

Every Clax object begins with a hidden, transpiler-managed **header** as its
first members, identical in every class. This common initial sequence is what
makes `Object`, `typeof`, and `clone` possible.

```c
// generated, shared by all classes
typedef enum { CLAX_TYPE_MyClass, CLAX_TYPE_Container, /* ... */ } ClaxType;

#define CLAX_MAGIC 0xC1A55EU

typedef struct ClaxHeader {
    unsigned int __magic;   // always CLAX_MAGIC
    ClaxType     __type;    // which class this object is
} ClaxHeader;
```

Each class struct places these two fields first:

```c
typedef struct MyClass {
    unsigned int __magic;   // CLAX_MAGIC
    ClaxType     __type;    // CLAX_TYPE_MyClass
    int x, y;               // user-declared members follow
} MyClass;
```

- The constructor writes `__magic` and `__type` before any user code runs (the
  transpiler injects this).
- Because the header is at offset 0 in every class, any object pointer — even an
  untyped `Object` — can be inspected for its magic and type.
- The `__magic` value guards against treating a non-Clax pointer as an object:
  operations that inspect an `Object` check `__magic` first and raise a runtime
  error (Section 12) if it does not match.

The header costs 8 bytes per object. This is the price of `Object` and
`typeof`; it is paid by every class uniformly.

---

## 8. The `Object` Type

`Object` is a built-in type usable as a method parameter type or return type. It
denotes a reference to **any** class instance.

```c
void add(Object other) {
    // 'other' may be an instance of any class
}
```

In generated C, `Object` is `void*`. Because objects share the header
(Section 7), an `Object` can always be queried for its real type with `typeof`.

### 8.1 Member and method access on `Object`

The concrete class of an `Object` is unknown at transpile time, so direct member
access (`other::x`) and direct method calls (`other::add(...)`) on an
`Object`-typed value are **only permitted after a `typeof` check that narrows
it**, within the guarded scope:

```c
void handle(Object other) {
    if (other::typeof(MyClass)) {
        // inside here, 'other' is known to be MyClass;
        // other::getX() and other::x are permitted
        int v = other::getX();
    }
}
```

The transpiler tracks that, inside the `true` branch of an
`other::typeof(SomeClass)` test, the variable `other` may be used as
`SomeClass`. Outside such a narrowed scope, only `typeof`, `clone`, assignment,
passing as `Object`, comparison with `NULL`, and `delete` are permitted on an
`Object`.

### 8.2 Dispatch

A narrowed access lowers to a direct mangled call for the narrowed class
(`MyClass__getX(other)`). Clax V2 does **not** provide implicit virtual
dispatch; calling a method on a non-narrowed `Object` is a transpile-time error.
This keeps the model simple and the generated C fully static. (A future version
may add automatic dispatch.)

---

## 9. `typeof`

`typeof` is built into every object and queried with `::`. It returns a C `bool`
(`<stdbool.h>`).

```c
bool b = obj::typeof(MyClass);
```

- `obj::typeof(SomeClass)` is `true` if `obj` is an instance of `SomeClass`.
- It works on normally-typed objects and on `Object`-typed values alike.
- On an `Object`, `typeof` also acts as a type-narrowing guard (Section 8.1).
- `NULL::typeof(...)` — i.e. `typeof` on a `NULL` object — is `false`.

Lowering: `obj::typeof(MyClass)` becomes a check of the header — magic valid and
`__type == CLAX_TYPE_MyClass`.

---

## 10. `clone`

`clone` is built into every object and produces an independent heap copy.

```c
MyClass copy = original::clone();
```

- Returns a new object of the same class as the receiver.
- The returned object is owned by the caller and must be `delete`d.
- **`clone` is a deep copy.** C-typed members are copied by value. Class-typed
  members (Section 6) are themselves cloned recursively, so the copy shares no
  objects with the original. A `NULL` class-typed member stays `NULL`.
- `clone` on a `NULL` object returns `NULL`.
- The transpiler generates a `ClassName__clone` function for every class.

Rationale: deep copy makes `clone` + `delete` always safe — deleting a clone can
never double-free anything owned by the original.

---

## 11. `NULL`

- Any object variable, parameter, member, or return value may be `NULL`.
- Class-typed members are `NULL` until assigned (Section 6).
- `delete` on a `NULL` object is a safe no-op.
- `typeof` and `clone` on a `NULL` object yield `false` and `NULL` respectively.
- Calling a user method or accessing a member of a `NULL` object is undefined
  behavior — a C null dereference — exactly as in C. The programmer guards with
  `if (obj != NULL)` where needed.

---

## 12. Errors

The transpiler distinguishes two error classes:

**Transpile-time errors** (reported by the transpiler, no C is emitted):

- `::` used on a non-class, undeclared, or non-narrowed `Object` variable.
- A method or member access on an `Object` outside a `typeof`-narrowed scope.
- `new` used outside a standalone single-variable declaration.
- A class file whose `class` name does not match its file name.
- A reference to a class that is not `#include`d.

**Runtime errors** (checked in generated C):

- Inspecting an `Object` whose `__magic` does not equal `CLAX_MAGIC` (a non-Clax
  pointer passed where an object was expected). The generated code reports the
  error and aborts.

## 13. Transpilation Model

Each class file transpiles to a **`.h` + `.c` pair**; each program file
transpiles to a `.c`. The `.h` holds the struct typedef and function
prototypes; the `.c` holds the function bodies. This avoids C multiple-definition
link errors and enables incremental builds.

All generated files — class pairs, program `.c`s, the shared
`clax_runtime.h`/`clax_runtime.c`, and the copy of `project.hx` named
`project.h` — are emitted into `<project-dir>/generated_c/`. Source `.cx`
files are never modified. `project.hx` is created in the project directory
on first run if absent.

### 13.1 Generated files for `MyClass.cx`

`MyClass.h`:

```c
#ifndef MYCLASS_H
#define MYCLASS_H
#include "clax_runtime.h"
typedef struct MyClass {
    unsigned int __magic;
    ClaxType     __type;
    int x, y;
} MyClass;
void     MyClass__ctor(MyClass* this, int x, int y);
void     MyClass__dtor(MyClass* this);
void     MyClass__add(MyClass* this, MyClass* other);
int      MyClass__getX(MyClass* this);
MyClass* MyClass__clone(MyClass* this);
#endif
```

`MyClass.c`:

```c
#include "MyClass.h"
#include <stdlib.h>
void MyClass__ctor(MyClass* this, int x, int y) {
    this->__magic = CLAX_MAGIC;
    this->__type  = CLAX_TYPE_MyClass;
    this->x = x;
    this->y = y;
}
void MyClass__dtor(MyClass* this) { }
void MyClass__add(MyClass* this, MyClass* other) {
    this->x += other->x;
    this->y += other->y;
}
int MyClass__getX(MyClass* this) { return this->x; }
MyClass* MyClass__clone(MyClass* this) {
    if (this == NULL) return NULL;
    MyClass* c = malloc(sizeof(MyClass));
    *c = *this;                 // copies header + C-typed members
    /* class-typed members would be deep-cloned here */
    return c;
}
```

### 13.2 Shared runtime — `clax_runtime.h` + `clax_runtime.c`

The transpiler emits a shared runtime header and source for the project. The
header (`clax_runtime.h`) declares:

- `CLAX_MAGIC` (`0xC1A55EU`),
- `ClaxType` — an enum with one entry per class actually used in the project,
- `ClaxHeader` — the common-initial-sequence struct `{__magic, __type}`,
- forward declarations of every class struct and its `Class__dtor` /
  `Class__clone` functions,
- the generic dispatch helpers `clax_delete(void*)` and `clax_clone(void*)`
  and the abort-on-misuse helper `clax_runtime_error(const char*)`.

The source (`clax_runtime.c`) defines `clax_delete` and `clax_clone` as
`switch`es on `ClaxHeader::__type` that forward to the per-class `__dtor` and
`__clone` functions. These are how `delete` and `clone` work on an
`Object`-typed receiver (Section 8) and how `delete` works on a class-typed
member expression like `delete this::field` (which the transpiler does not
know the static type of at the call site).

### 13.3 Transformation rules

| Clax | Generated C |
|------|-------------|
| `class C { ... }` | `struct C` typedef (header fields first) + mangled functions |
| data member | struct field |
| `C(...)` constructor | `void C__ctor(C* this, ...)`, header writes injected first |
| `~C()` destructor | `void C__dtor(C* this)` |
| `ret m(...)` | `ret C__m(C* this, ...)` |
| `this::member` | `this->member` |
| `obj::member` | `obj->member` |
| `obj::method(a)` | `C__method(obj, a)` |
| `obj::typeof(T)` | header magic check + `obj->__type == CLAX_TYPE_T` |
| `obj::clone()` | `C__clone(obj)` |
| `callable(obj::method)` | `&C__method__cb` (address of the generated trampoline) |
| `C obj = new C(a);` | `C* obj = malloc(sizeof(C)); C__ctor(obj, a);` |
| `delete obj;` (statically-typed `C`) | `do { if (obj) { C__dtor(obj); free(obj); } } while (0);` |
| `delete EXPR;` (other forms, incl. `Object`, member expr) | `clax_delete(EXPR);` |
| `obj::clone()` on `Object` | `clax_clone(obj)` |
| `Object` (param/return) | `void*` |
| `#include "./C.cx"` | `#include "C.h"` |
| `#include <C.cx>` (resolved via `--include`) | `#include "C.h"` |
| any other C code | unchanged |

### 13.4 Name mangling

Class functions are `ClassName__memberName`. Constructor: `ClassName__ctor`.
Destructor: `ClassName__dtor`. Built-in: `ClassName__clone`. Every method takes
`ClassName* this` first.

A method that declares an `object` parameter also gets a callback trampoline
`ClassName__memberName__cb` (Section 3.5): same parameter list as the method but
**without** the `this`, forwarding to the method with `(ClassName*)object` as the
receiver. It is a non-`static` function, prototyped in the class header so
`callable()` can reference it from any file.

### 13.5 Symbol tracking

The transpiler maintains a per-scope table mapping object variables to their
class, populated from:

- `C name = new C(...);`
- a class-typed parameter `C name`
- a class-typed member `C name;` (mapped on `this::name`)
- the narrowed variable inside an `obj::typeof(C)` true-branch

`Object`-typed variables are tracked as "unknown class" until narrowed.

## 14. Build Process

The Clax transpiler is a **single C-language command**:

```
clax [--lint] [--init-project] [--include=DIR ...] <project-dir>
```

- `<project-dir>` — directory containing the project's own `.cx` (and any
  `.c`) files. The output directory `<project-dir>/generated_c/` is created if
  it does not already exist.
- `--include=DIR` — register `DIR` as a search path for `#include <X.cx>`
  resolution. May be given multiple times; directories are searched in the
  order given.
- `--lint` — validate the project without writing any files. Returns exit 0
  and prints `lint OK` on success; returns non-zero with a precise error
  message on any spec violation or unresolved include. Useful as a pre-commit
  check or editor diagnostic.
- `--init-project` — scaffold a new project at `<project-dir>` and exit. The
  resulting layout is:

  ```
  <project-dir>/
    build.sh              # transpile + compile (outside the project subdir)
    project/
      main.cx             # hello-world entry point
      project.hx          # project-wide constants
  ```

  `<project-dir>/project` is the actual Clax project directory consumed by
  the transpiler; everything the user adds (more `.cx` files, edits to
  `project.hx`) goes in there. `build.sh` lives one level up so the produced
  binary and generated tree don't sit alongside the source. The generated
  `build.sh` invokes `clax` as an unqualified command (assuming it is on
  `PATH`) and bakes in an absolute path to the shared `system/` directory.
  Both are overridable via the `CLAX` and `SYSTEM` env vars (along with
  `CC`, `CFLAGS`, `LDFLAGS`). The command refuses to overwrite an existing
  `build.sh` or `project/main.cx`.

A `clax_lint` symlink to the `clax` binary is provided as a convenience: it
invokes the same binary with lint mode auto-enabled (detected via `argv[0]`).

A build is two passes:

1. **Transpile.** Run the Clax transpiler over the project directory. The
   transpiler:
   - Tokenizes every `.cx` in the project directory.
   - Resolves every `.cx` include (quoted or angle-bracketed) and recursively
     pulls in transitively-referenced classes from `--include` directories.
   - Collects all class names in a first pass.
   - In a second pass, emits each class `.cx` as a `.h`+`.c` pair and each
     program `.cx` as a `.c`, all into `<project-dir>/generated_c/`.
   - Emits the shared `clax_runtime.h`+`clax_runtime.c` into the same place.
2. **Compile and link.** Compile every `.c` in `generated_c/` with a normal
   C compiler and link them into an executable.

Example shell flow:

```sh
clax --include=./system ./project
gcc -O2 -o project/program project/generated_c/*.c
```

The C compiler is invoked once over the whole `generated_c/` directory; there
are no Clax-specific link rules.

## 15. Complete Example

### `MyClass.cx`

```c
class MyClass {
    int x, y;
    MyClass(int x, int y) {
        this::x = x;
        this::y = y;
    }
    ~MyClass() { }
    void add(MyClass other) {
        this::x += other::x;
        this::y += other::y;
    }
    int getX() { return this::x; }
};
```

### `main.cx`

```c
#include "./MyClass.cx"
#include <stdio.h>

void describe(Object o) {
    if (o::typeof(MyClass)) {
        printf("MyClass with x = %d\n", o::getX());
    }
}

int main(void) {
    MyClass a = new MyClass(1, 2);
    MyClass b = new MyClass(10, 20);

    a::add(b);
    printf("%d\n", a::getX());     // 11

    MyClass c = a::clone();        // independent copy, caller owns it
    describe(c);                   // narrowed via typeof

    delete a;
    delete b;
    delete c;
    return 0;
}
```

### Generated `main.c` (abridged)

```c
#include "MyClass.h"
#include <stdio.h>

void describe(void* o) {
    if (o && ((MyClass*)o)->__magic == CLAX_MAGIC
          && ((MyClass*)o)->__type == CLAX_TYPE_MyClass) {
        printf("MyClass with x = %d\n", MyClass__getX((MyClass*)o));
    }
}

int main(void) {
    MyClass* a = malloc(sizeof(MyClass)); MyClass__ctor(a, 1, 2);
    MyClass* b = malloc(sizeof(MyClass)); MyClass__ctor(b, 10, 20);

    MyClass__add(a, b);
    printf("%d\n", MyClass__getX(a));

    MyClass* c = MyClass__clone(a);
    describe(c);

    if (a) { MyClass__dtor(a); free(a); }
    if (b) { MyClass__dtor(b); free(b); }
    if (c) { MyClass__dtor(c); free(c); }
    return 0;
}
```

## 16. Changes from V1

- File extension changed from `.cls` to `.cx`.
- Classes may now contain class-typed members (as references; Section 6).
- Methods may now return objects (V1 restricted returns to C types).
- Object parameters are written as `C other` (a reference), not `C* other`.
- Added the `Object` type for accepting any class (Section 8).
- Added built-in `typeof` (Section 9) and `clone` (Section 10).
- Added `callable()` for passing a method as a C function-pointer callback,
  via a generated trampoline (Section 3.5).
- Objects may be `NULL` (Section 11).
- Every object now carries a hidden header (magic + type tag; Section 7).
- The transpiler emits a shared `clax_runtime.h` and `clax_runtime.c`. The
  runtime source provides generic `clax_delete` and `clax_clone` dispatchers
  used for `Object`-typed receivers and for `delete` on class-member
  expressions whose static type is unknown at the call site.
- The transpiler is specified as a single C-language command operating on a
  single project directory, with all generated C placed in
  `<project-dir>/generated_c/`.
- Reusable classes may live outside the project directory and be brought in
  with `--include=DIR` search paths; class files reference them with
  `#include <X.cx>` (Section 2.1).
- A `--lint` mode (also exposed as the `clax_lint` symlink) runs the full
  validation pipeline without writing any files.
- Each project has a single hand-written header, `project.hx` (Section 2.2),
  for project-wide constants. No other `.h` files are permitted in a project;
  all other headers are transpiler-generated.
- `--init-project` scaffolds a new Clax project in one command (Section 14).
